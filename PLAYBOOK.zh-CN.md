# Bridge MCP 玩法手册（Playbook）— AI 工作群配方

这份手册提供一组"可复现、可照做"的玩法，让你用 Bridge MCP 快速搭起多 Agent 的"AI 工作群"。

默认你已经全局注册：
```bash
pip install mcp
claude mcp add bridge --scope=user "python" "C:/ccbridge/bridge.py"
```

## 开始前 3 条简单规则
1. **默认组织形态：** 一个主管 + 多个员工。你只管主管，主管去指挥其他人。
2. **员工保持待命监听：** 跑 `recv(86400)`，收到消息就快速回复，然后立刻恢复监听。
3. **输出尽量短 + 结构化：** 一行状态、编号要点、可验收清单，少废话。

**注意：** Agent ID 是随机分配的。发私聊前先用 `get_status()` 看实际 ID。

## 玩法 1：总经理 + 员工（最经典的"派活→汇报→总结"）
**目标：** 主管分配任务；员工并行执行；主管收汇报并决策。

### 搭建
- **终端 A：** 总经理（主管）
- **终端 B：** 员工 1
- **终端 C：** 员工 2

### 员工指令（复制粘贴）
对员工 1、员工 2 说：
> "你保持持续监听；收到任何信息都立刻用简短结构化方式回复；回复完立刻恢复监听。"

然后每个员工执行：
```bash
recv(86400)
```

### 主管流程
1) 先看在线与 ID：
```bash
get_status()
```

2) 派活（私聊更稳，也可群发）：
```bash
send("<员工1ID>", "任务：研究方案A。回复：步骤+取舍+推荐。然后继续监听。")
send("<员工2ID>", "任务：研究方案B。回复：风险+成本+时间线。然后继续监听。")
```

3) 收汇报：
```bash
recv(120)
```

4) 主管总结：合并两份结论，给最终方案与下一步。

## 玩法 2：部门制（每个文件夹一个身份，超持久上下文）
**目标：** 像开公司一样，把 Agent 变成长期员工。

### 组织方式
每个 Agent 在自己的文件夹运行：
- `company/CEO/`
- `company/Research/`
- `company/Engineering/`
- `company/QA/`

每个文件夹放自己的 `CLAUDE.md`（或记忆文件）→ 身份稳定、上下文很持久。

### 最小闭环
CEO 派活 → 部门汇报 → QA 审计 → CEO 决策与复盘

**提示：** 把"文件夹"当员工，而不是当一次性任务。

## 玩法 3：私聊干活，all 求救（降低你盯群成本）
**规则：**
- 员工之间用私聊协作、推进任务
- 卡住就群里喊一句：
```bash
send("all", "我卡在X，谁给我3步解决方案？")
```
- 主管只在出现求救/冲突时介入。

## 玩法 4：研究冲刺（并行出方案 → 主管选最优）
**目标：** 快速拿到多套独立答案。

### 主管群发：
```bash
send("all", "冲刺：请给3种方案。回复格式：1)方案 2)风险 3)快赢。然后继续监听。")
recv(90)
```

### 后续：
指定一人深挖、一人挑错、一人做验收清单。

## 玩法 5：审计员 Agent（复盘、纠错、兜底）
**目标：** 长期迭代质量更高。

### 角色
- 主管（CEO）
- 员工（执行）
- 审计员（QA）

### 审计员要固定检查：
假设、边界、失败模式、最小复现、验收步骤。

### 主管可以这样派审计：
```bash
send("<QA_ID>", "审计员工输出：给我5个风险+5个修复+最终结论。")
recv(120)
```

## 玩法 6：一句话让 AI 管 AI（核心特色）
一条群发就能组织很多人：
```bash
send("all", "<001-003> 研究；<004-006> 起草；<007-009> 待命。只回复ID+一句状态。")
```

**说明：** 分工协作是提示词/流程的范畴；Bridge MCP 提供稳定低成本通信底座。

## 玩法 7：混编团队（理论）
理论上任何支持 MCP 的客户端都能加入同一"工作群"。
如果你确认可用，就把它当另一个员工：给它一个角色文件夹+角色提示词+待命监听。

## 玩法 8：长任务（不一定省 token，但更省人工迭代）
并行不必然降低模型 token，但通常能显著降低你的人工成本：
- 更快拿到多份反馈
- 更容易追问与复盘
- 更快进入"可验收"的最终版本

## 玩法 9：协作游戏（娱乐/压力测试）
用合作类"牌/棋/回合制"模拟（不涉及金钱/赌博）测试：
- 一人当裁判维护状态
- 其他人提议行动

测试多 Agent 的轮转、沟通清晰度与冲突处理。

## 常见问题排查
- **群发后没人回：** 再 `recv(30)` 一次（轮询近实时）。
- **DB Busy/Locked：** 关掉 DB 查看器，避免同步盘目录，减少同时写入。
- **无法"打断 recv"：** 用短轮询模式：反复 `recv(5)`。
